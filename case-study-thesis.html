LLM Order Extraction
Thesis

研究大型語言模型於異質訂單文件之關鍵資訊擷取與標準化，建立從文件解析、Prompt 設計到評估分析的完整可重現流程，Level 2 正式結果達 95.2% Document Accuracy。

Python · LLM · Prompt Engineering 學術研究 / 畢業論文 95.2% Document Accuracy
01 — 專案背景

為什麼要研究訂單資訊抽取？

企業每天收到大量來自不同客戶、不同格式的訂單文件——有 PDF、有 Excel、有掃描圖片，每份的欄位命名、排版、語言都不同。要把這些訂單「讀懂」並整理成系統能用的結構化資料，傳統方法是人工鍵入，或是用規則型 OCR（非常不靈活，每換一個格式就需要重新設定）。

這篇論文的核心命題是：能不能用 LLM 來做到格式無關的訂單資訊抽取，同時讓結果是結構化、可評估、可重現的？

研究目標：建立一套 LLM-based 的訂單抽取 Pipeline，輸入任意格式訂單 → 輸出標準化 JSON，並提出可量化的評估指標體系。
02 — 問題與挑戰

異質訂單文件的核心難題

難題 1欄位名稱多變：同一欄位在不同訂單叫「數量」、「Qty」、「Q'ty」、「pcs」，語意相同但字面完全不同，規則型方法幾乎無法覆蓋。
難題 2資訊分散與干擾：訂單中常夾雜備註、免責聲明、頁首頁尾等無關文字，模型需要精準找到有效資訊並忽略噪音。
難題 3評估困難：LLM 輸出是文字，怎麼定義「正確」？如何量化比較不同 Prompt 的效果？需要自行設計評估框架。
難題 4格式一致性：即使抽取正確，輸出格式若不固定，下游系統就無法直接串接，需要強制結構化輸出。
03 — 解法與技術架構

三層 Pipeline + JSON Schema 強制結構化

📄
文件解析層→PDF 文字抽取 / OCR 掃描圖片（Level 3），將原始文件轉成純文字
🧠
LLM 抽取層→Prompt Engineering + JSON Schema 指定輸出格式，呼叫 OpenAI API
📊
評估層→欄位級別 / 文件級別精確率，多組 Prompt 對比分析

關鍵設計決策是使用 JSON Schema 作為 LLM 的輸出約束——明確告訴模型「你只能回傳這個格式的 JSON，每個欄位的型別與允許值如下」，大幅降低輸出格式不一致的問題。

// JSON Schema 片段（部分欄位） { "type": "object", "properties": { "po_number": { "type": "string" }, "order_date": { "type": "string", "format": "date" }, "line_items": { "type": "array", "items": { ... } }, "total_amount": { "type": "number" } }, "required": ["po_number", "line_items"] }

04 — 我的負責範圍

從研究設計到實作評估，獨立完成

  • 1文獻回顧——整理 LLM 資訊抽取、IE（Information Extraction）與 OCR 相關研究背景
  • 2收集並標注真實異質訂單資料集，建立 Ground Truth 標注規範
  • 3設計三個評估層級（Level 1~3）與對應評估指標：Field-level、Document-level Accuracy
  • 4Prompt 工程實驗——設計多組 Prompt 策略（Zero-shot、Few-shot、Chain-of-thought），對比效果
  • 5建立自動化評估腳本，批次跑測試集並輸出比較報告
  • 6延伸至 Level 3 OCR 版本，處理掃描圖片訂單，整合 PaddleOCR
05 — 實作重點與難點

讓研究「可重現、可比較」是最難的部分

技術重點評估框架設計：自定義 Field-level Accuracy（每個欄位對不對）與 Document-level Accuracy（整份訂單全對才算對），讓不同 Prompt 策略的效果可以客觀比較。
技術重點Prompt 控制變數：同一份訂單、同一個模型，只改 Prompt 寫法，確保實驗結果差異是來自 Prompt 而非其他因素，建立嚴謹的對照組設計。
技術重點OCR 誤字修正：Level 3 的 OCR 輸出常有誤字（數字 0 / O 混淆、中文斷字等），需要在 LLM Prompt 中加入糾錯指令並在後處理層做驗證。

研究成果不只是一個數字——更重要的是建立了一套可重現、可量化的評估體系，讓後續研究者能在同一基準上比較不同方法的效果。輸出的標準 JSON 可直接串接 ERP 或訂單管理系統，具備實際落地價值。

企業應用延伸：這套框架可以直接應用於企業的文件自動化場景——合約資訊抽取、採購單解析、客服紀錄結構化等，不需要重新設計，只需調整 JSON Schema 的欄位定義。
