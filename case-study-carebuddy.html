LINE Bot / Webhook

CareBuddy
LINE 健康陪伴機器人

透過 LINE Webhook 架構打造的健康陪伴機器人，讓使用者直接在 LINE 聊天視窗完成健康紀錄的新增、查詢、更新與刪除（CRUD），並透過溫暖的對話設計讓健康管理更有人情味。

LINE Bot · Webhook · Python 個人作品 / Open Source CRUD + 健康紀錄 + 陪伴感
01 — 專案背景

把健康管理放進你每天都會用的 App

大多數人不會專門下載健康管理 App 來記錄身體狀況，因為要另外開 App 本身就是一個門檻。但 LINE 是台灣使用率最高的通訊軟體，幾乎人人每天都會開——如果健康記錄可以直接在 LINE 裡完成，使用頻率就會大幅提升。

CareBuddy 的設計核心不是「功能盡量多」，而是「讓使用者願意每天來說話」。除了基本的健康紀錄功能，機器人的回應語氣被刻意設計成有溫度、像朋友的風格，而不是冷冰冰的資料庫介面。

設計理念：健康管理工具最大的問題不是功能不夠，而是沒有人想每天打開它。CareBuddy 選擇待在使用者已經習慣的地方——LINE。

03 — 問題與痛點

健康管理工具為什麼都沒人用

痛點 1需要另外下載 App——使用門檻太高，很多人安裝後用一次就忘了。
痛點 2介面太冰冷——填表格式的輸入讓「記錄健康」感覺像在填作業，而不是在照顧自己。
痛點 3沒有回饋感——記下去之後什麼都沒有，沒有鼓勵、沒有提醒，慢慢就不想記了。
痛點 4操作不直覺——要找到「新增」在哪裡、「查詢」在哪裡，學習成本讓人放棄。
04 — 我的負責範圍

從架構設計到部署，個人獨立完成

  • 1設計 LINE Bot 的整體對話流程，規劃各種使用情境的回應邏輯
  • 2建立 Webhook Server——使用 Python 接收並解析 LINE 事件，路由到對應處理器
  • 3設計資料庫 Schema——健康紀錄表、使用者資料表，支援多種紀錄類型
  • 4實作 CRUD 完整邏輯——新增、查詢、更新、刪除，含錯誤處理與邊界情況
  • 5設計意圖辨識層——從自然語言輸入判斷使用者想做什麼操作
  • 6撰寫有溫度的回應模板，讓機器人感覺像在陪伴而非只是在執行指令
05 — 實作重點與難點

讓機器人有「陪伴感」的技術決策

技術重點自然語言意圖辨識：使用者不會說「/add 體重 65」，他們會說「我今天 65 公斤」或「剛量了體重，65」。需要設計能理解這種非結構化輸入的意圖辨識層。
技術重點多使用者資料隔離：LINE Bot 可能同時有很多人在用，需要確保每個人只能看到自己的紀錄。以 LINE user_id 作為隔離 key，所有查詢都加上 user_id 過濾條件。
技術重點Webhook 即時性：LINE 的 Webhook 要求伺服器必須在一定時間內回應，否則使用者會感覺機器人沒反應。需要設計非同步處理架構，避免資料庫操作阻塞回應。
