AI Nutritionist

整合熱量分析、圖片辨識熱量與 AI 飲食教練三大功能，把分散的健康查詢工具收進同一套系統，讓日常健康管理的門檻大幅降低。

Python · LLM · RAG 個人作品 / Open Source 三合一健康管理
01 — 專案背景

為什麼要做這個工具？

想管理飲食健康，通常要用到好幾個工具：查熱量用一個 App、拍照估熱量要另一個、想問「我這樣吃合理嗎」又要去找別的地方。這些工具各自為政，沒有辦法形成一個完整的使用流程。

這個專案的出發點是：把最常用的三個健康查詢動作整合在一個系統裡，讓使用者不需要切換工具，從查詢、記錄到獲得建議都在同一個介面完成。

設計理念：不是做一個功能很多但沒人用完的 App，而是聚焦在「每天真的會用到的三個動作」上，做到每個功能都夠好用。
02 — 問題與痛點

健康管理工具的使用斷點在哪裡

痛點 1熱量資料散落各地——要查一個食物的熱量，可能要搜尋好幾個網站或 App 才找到可信的數字。
痛點 2拍照估熱量的需求很強，但多數工具要另外下載、另外登入，使用門檻高，用一次就不想用第二次。
痛點 3想問「我今天這樣吃熱量夠嗎」、「減脂期要怎麼配比」，但沒有一個工具能回答個人化的飲食問題。
痛點 4沒有整合——查完熱量、拍完照片、問完問題，資料都在不同地方，無法形成完整的飲食紀錄。
03 — 功能與技術架構

三大核心功能模組

🔍

文字熱量查詢

輸入食物名稱，整合營養資料庫與 LLM，快速返回熱量、蛋白質、脂肪、碳水等完整數據。

📸

圖片熱量辨識

上傳餐點照片，AI 自動辨識食物種類並估算熱量，不需要逐一手動查詢。

結合 RAG 知識庫，根據使用者的飲食目標與紀錄，給出個人化的飲食建議與規劃方向。

04 — 我的負責範圍

從需求定義到功能實作，個人獨立開發

  • 1確認三大核心功能範疇與使用者流程設計
  • 2建立營養資訊 RAG 知識庫——收集、清理、向量化營養學資料
  • 3設計文字查詢模組的 Prompt，確保輸出格式一致（熱量、三大營養素、份量說明）
  • 4實作圖片辨識模組，整合 Vision API 並設計提示詞引導準確估算
  • 5建立 AI 飲食教練的對話邏輯，支援多輪對話與根據歷史紀錄給建議
  • 6整合三個模組為統一介面，上傳至 GitHub 開源
05 — 實作重點與難點

讓三個功能「感覺像同一個系統」

技術重點圖片估算的準確性：圖片辨識熱量本質上是估算，關鍵在 Prompt 設計——要引導模型先辨識食物種類、再估份量、再算熱量，分步驟比直接問答準確得多。
技術重點RAG 知識庫的邊界設定：飲食建議很容易被使用者導到「醫療建議」，需要在 Prompt 中明確設定邊界——建議飲食方向可以，但特定疾病的飲食治療要引導就醫。
技術重點跨模組資料共享：三個功能需要共享使用者的飲食紀錄，這樣 AI 教練才能根據「你今天已經吃了什麼」給個人化建議，需要設計共用的 session 狀態管理。

把原本需要跨多個 App 才能完成的健康管理流程，收進同一套系統。使用者不再需要在多個工具之間切換，也不需要手動彙整資料——系統自動保存紀錄並作為後續建議的依據。

延伸可能：這套架構可以很容易地擴展成企業的員工健康管理平台，或是結合 LINE Bot 讓使用者在聊天視窗就能完成所有功能，不需要另外開 App。
